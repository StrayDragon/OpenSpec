通过创建下一个产物来继续进行变更。

**输入**：可选指定变更名称。如果省略，必须提示选择可用变更。

**步骤**

1. **如果没有提供变更名称，提示选择**

   运行 `openspec list --json` 获取按最近修改时间排序的可用变更。然后使用 **AskUserQuestion 工具** 让用户选择要处理的变更。

   展示最近修改的 3-4 个变更作为选项，显示：
   - 变更名称
   - 状态（例如，"0/5 任务"、"完成"、"无任务"）
   - 最近修改时间（来自 `lastModified` 字段）

   将最近修改的变更标记为"（推荐）"，因为它可能是用户想要继续的。

   **重要**：不要猜测或自动选择变更。始终让用户选择。

2. **检查当前状态**
   ```bash
   openspec status --change "<name>" --json
   ```
   解析 JSON 以了解当前状态。

3. **根据状态行动**：

   ---

   **如果所有产物都已完成（`isComplete: true`）**：
   - 祝贺用户
   - 显示最终状态
   - 建议："所有产物已创建！现在可以实施此变更或将其归档。"
   - 停止

   ---

   **如果有产物准备好创建**（状态显示 `status: "ready"` 的产物）：
   - 从状态输出中选择第一个 `status: "ready"` 的产物
   - 获取其指令：
     ```bash
     openspec instructions <artifact-id> --change "<name>" --json
     ```
   - 解析 JSON 以获取模板、依赖关系及其解锁的内容
   - **使用模板作为起点创建产物文件**：
     - 阅读任何已完成的依赖文件以获取上下文
     - 根据上下文和用户的目标填充模板
     - 写入指令中指定的输出路径
   - 显示创建了什么以及现在解锁了什么
   - 创建**一个**产物后停止

   ---

   **如果没有产物准备好创建（全部被阻塞）**：
   - 这不应该在有效的规范中发生
   - 显示状态并建议检查问题

4. **创建产物后，显示进度**
   ```bash
   openspec status --change "<name>"
   ```

**输出**

每次调用后显示：
- 创建了哪个产物
- 当前进度（N/M 完成）
- 现在解锁了哪些产物
- 提示："想继续吗？只需让我继续或告诉我下一步要做什么。"

**产物创建指南**

填充模板时：

- **proposal.md**：如果不清晰，询问用户关于变更的内容。填充"为什么"、"变更内容"、"能力"、"影响"。
  - **重要**：能力部分很关键。在填充之前：
    - 检查 `openspec/specs/` 中现有能力
    - 列出具有短横线分隔名称的新能力（例如 `user-auth`、`data-export`）
    - 列出需要规范更新的已修改能力
  - 列出的每个能力在下一阶段都需要相应的规范文件。
- **specs/*.md**：根据提案中列出的每个能力创建一个规范。使用 `specs/<capability-name>/spec.md` 路径。
- **design.md**：记录技术决策、架构和实施方法。
- **tasks.md**：根据规范和设计将实施分解为带复选框的任务。

**护栏**
- 每次调用创建一个产物
- 在创建新产物之前始终阅读依赖产物
- 不要跳过产物或无序创建
- 如果上下文不清楚，在创建之前询问用户
- 写入后验证产物文件是否存在，然后才标记进度
