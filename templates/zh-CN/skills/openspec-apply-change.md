实现 OpenSpec 变更中的任务。

**输入**：可选指定变更名称。如果省略，必须提示可用变更。

**步骤**

1. **若未提供变更名称，提示选择**

   运行 `openspec list --json` 获取可用变更，使用 **AskUserQuestion tool** 让用户选择。

   展示包含 tasks.md 的变更（可实施）。
   将任务未完成的变更标记为 "（进行中）"。

   **重要**：不要猜测或自动选择变更，必须让用户选择。

2. **获取 apply 指令**

   ```bash
   openspec instructions apply --change "<name>" --json
   ```

   该命令返回：
   - 上下文文件路径（proposal、specs、design、tasks）
   - 进度（总数、完成、剩余）
   - 任务列表及状态
   - 基于当前状态的动态指令

   **状态处理：**
   - 若 `state: "blocked"`（缺少产物）：展示消息并建议使用 openspec-continue-change
   - 若 `state: "all_done"`：祝贺并建议归档
   - 否则：进入实现流程

3. **读取上下文文件**

   读取 `contextFiles` 中列出的文件：
   - `proposal` - 为什么、做什么
   - `specs` - 需求和场景（使用 glob 找到全部）
   - `design` - 技术方案（如存在）
   - `tasks` - 实现清单

4. **展示当前进度**

   展示：
   - 进度："N/M tasks complete"
   - 剩余任务概览
   - CLI 返回的动态指令

5. **实现任务（循环直到完成或阻塞）**

   对每个待办任务：
   - 标明正在处理的任务
   - 完成必要的代码变更
   - 保持改动最小且聚焦
   - 在 tasks.md 中标记完成：`- [ ]` → `- [x]`
   - 继续下一个任务

   **暂停条件：**
   - 任务不清晰 → 询问澄清
   - 实现中发现设计问题 → 建议更新产物
   - 遇到错误或阻塞 → 汇报并等待指导
   - 用户中断

6. **完成或暂停时，展示状态**

   展示：
   - 本次完成的任务
   - 总体进度："N/M tasks complete"
   - 若全部完成：建议归档
   - 若暂停：说明原因并等待指导

**实现过程输出**

```
## 实现中: <change-name>

正在处理任务 3/7: <task description>
[...正在实现...]
✓ 任务完成

正在处理任务 4/7: <task description>
[...正在实现...]
✓ 任务完成
```

**完成时输出**

```
## 实现完成

**变更:** <change-name>
**进度:** 7/7 tasks complete ✓

### 本次完成
- [x] Task 1
- [x] Task 2
...

所有任务已完成！可以归档此变更。
```

**暂停时输出（遇到问题）**

```
## 实现暂停

**变更:** <change-name>
**进度:** 4/7 tasks complete

### 遇到的问题
<问题描述>

**选项:**
1. <选项 1>
2. <选项 2>
3. 其他方案

你希望怎么做？
```

**约束**
- 按任务推进，直到完成或被阻塞
- 开始前必须读取上下文（specs、design）
- 若任务含糊，先暂停并询问再实现
- 若实现暴露问题，先暂停并建议更新产物
- 保持代码变更最小，聚焦当前任务
- 完成任务后立即更新复选框
- 遇到错误、阻塞或需求不清时暂停，不要猜测

**流式流程集成**

该技能支持“对变更执行动作”的模型：

- **可随时调用**：产物未完成但已有 tasks.md 时、部分实现后、与其他操作交错
- **允许更新产物**：实现中发现设计问题时可建议更新产物，不是阶段锁定
