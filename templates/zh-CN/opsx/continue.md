通过创建下一个产物继续推进变更。

**输入**：可在 `/opsx:continue` 后指定 `--change <name>`。如果省略，必须提示可用变更。

**步骤**

1. **若未提供变更名称，提示选择**

   运行 `openspec list --json` 获取按最近修改排序的可用变更，然后使用 **AskUserQuestion tool** 让用户选择要继续的变更。

   展示最近修改的 3-4 个变更选项，包含：
   - 变更名称
   - 状态（如 "0/5 tasks"、"complete"、"no tasks"）
   - 最近修改时间（来自 `lastModified` 字段）

   将最近修改的变更标记为 "（推荐）"，因为最可能是用户要继续的。

   **重要**：不要猜测或自动选择变更，必须让用户选择。

2. **检查当前状态**
   ```bash
   openspec status --change "<name>" --json
   ```
   解析 JSON 以了解当前状态。

3. **根据状态执行**：

   ---

   **如果所有产物都完成（`isComplete: true`）**：
   - 祝贺用户
   - 展示最终状态
   - 建议："所有产物已创建！你现在可以实现此变更或归档它。"
   - 停止

   ---

   **如果有产物可以创建**（状态显示 `status: "ready"`）：
   - 从状态输出中选择第一个 `status: "ready"` 的产物
   - 获取其说明：
     ```bash
     openspec instructions <artifact-id> --change "<name>" --json
     ```
   - 解析 JSON 获取模板、依赖和解锁内容
   - **创建产物文件**，以模板为起点：
     - 读取已完成的依赖文件作为上下文
     - 基于上下文与用户目标填充模板
     - 写入说明中指定的输出路径
   - 说明创建了什么以及现在解锁了什么
   - 创建 **一个** 产物后停止

   ---

   **如果没有产物就绪（全部阻塞）**：
   - 这在有效 schema 下不应发生
   - 展示状态并建议排查问题

4. **创建产物后展示进度**
   ```bash
   openspec status --change "<name>"
   ```

**输出**

每次调用后展示：
- 创建了哪个产物
- 当前进度（N/M 完成）
- 现在解锁了哪些产物
- 提示语："运行 `/opsx:continue` 创建下一个产物"

**产物创建指南**

填充模板时：

- **proposal.md**：如果变更不清楚，先询问用户。填写 Why、What Changes、Capabilities、Impact。
  - **重要**：Capabilities 部分很关键。在填写前：
    - 检查 `openspec/specs/` 中已有能力
    - 用 kebab-case 列出新能力（例如 `user-auth`、`data-export`）
    - 列出需要规格更新的已修改能力
  - 列出的每个能力都需要在下一阶段生成对应的规格文件。
- **specs/*.md**：为提案中列出的每个能力创建一个规格文件，路径为 `specs/<capability-name>/spec.md`。
- **design.md**：记录技术决策、架构与实现方案。
- **tasks.md**：基于规格和设计拆分为带复选框的实现任务。

**约束**
- 每次只创建 **一个** 产物
- 创建新产物前始终读取依赖产物
- 不要跳过产物或乱序创建
- 如上下文不清晰，先询问用户再创建
- 写入后确认产物文件存在，再标记进度
